# Security Policy

## Supported Versions

ARCH is released through GitHub tags and releases. Security fixes are shipped in the latest release.

| Version | Supported |
| --- | --- |
| Latest release | Yes |
| Older releases | Best effort |

## Reporting A Vulnerability

Please report security issues privately instead of opening a public issue.

Use GitHub private vulnerability reporting when available for this repository. If it is not available, contact the repository owner through GitHub and include:

- affected ARCH version or commit
- affected file or workflow
- impact and expected security boundary
- reproduction steps or proof-of-concept, if safe to share
- whether the issue is already public

Do not include real secrets, access tokens, or private project data in the report.

## Response Targets

- Initial triage: 7 days
- Fix or mitigation plan for valid issues: 30 days
- Public advisory or release notes after a fix is available, when appropriate

## Security Scope

In scope:

- ARCH skill instructions and templates
- local file writes performed by ARCH scripts
- Codex skill install and update script behavior
- repository validation and eval scripts
- GitHub Actions workflow and release process
- public install and update instructions

Out of scope:

- downstream applications generated by coding assistants after ARCH creates `context/`
- user project code that ARCH did not create
- social engineering, spam, or denial-of-service against GitHub

## Release Security

Before publishing a security-sensitive release, run the validation commands in `docs/release-process.md` and confirm the GitHub Actions workflow passes for both `main` and the release tag.
